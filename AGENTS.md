# This is NOT the Next.js you know

This repository uses Next.js 16. APIs, conventions, caching, and file structure may differ from
training data. Before changing Next.js behavior, read the relevant guide in
`node_modules/next/dist/docs/`. Follow current deprecations and use `proxy.ts`, not legacy
middleware conventions.
<!-- END:nextjs-agent-rules -->

BeerBoard is a beer intelligence and comparison product built around **Beers**, **Countries**,
**Styles**, **Rankings**, **Comparisons**, public user **Reviews**, private **Saved beers**, and transparent
methodology. It is a welcoming global guide, not an analyst dashboard.

Public browsing must work without an account. Authentication is required only for owner-specific
state such as publishing reviews, adding or removing saved beers, account settings, and future genuinely user-owned
features. Supabase is still required as the production catalog source.

The product name is **BeerBoard**. Treat any conflicting name in early screenshots as obsolete
unless a later repository decision explicitly changes the name.

## Non-negotiables

- Supabase Postgres is the production source of truth. Do not create a competing production beer
  catalog in TypeScript. Test fixtures and repeatable seed data are allowed.
- The canonical score is generated by PostgreSQL. TypeScript and React must not recalculate it.
- Every exposed table must have RLS before feature work is complete. Public catalog data is
  read-only to browser roles. User-owned rows are restricted by `auth.uid()`.
- Use `auth.getUser()` for server authorization. Never authorize from an unverified session or from
  `user_metadata`.
- Never expose a service-role or secret key to client code, browser bundles, logs, shared modules,
  proxy code, or `NEXT_PUBLIC_*` variables.
- Do not read signed-in state in `src/app/layout.tsx`. Keep public routes cacheable and resolve
  account chrome through a focused boundary.
- A migration file that has not been applied and verified is incomplete work. Keep generated
  database types synchronized with the deployed schema.
- Vercel is the required host. Never claim deployment success without a live URL and a successful
  deployment status.
- The current catalog is fictional demonstration data. Keep that disclosure visible wherever a
  ranking, score, freshness date, or aggregate could be mistaken for a verified real-world claim.
- Every visible control must work. Remove or complete decorative search, filter, export, compare,
  account, settings, review, and saved-beer controls.

## Screenshot capability contract

The original screenshots are raw functional references. They are not approved designs. Functional
parity means retaining and improving the capabilities, not copying typography, spacing, or a
desktop-only composition.

The following capabilities are non-negotiable:

- Ranking: beer ranking, search, country filter, style filter, minimum-score filter, apply, clear,
  CSV export, pagination or progressive loading, and data freshness.
- Beer details: profile routes, technical measurements, Index score, editorial verdict, sensory
  visualization with exact accessible values, dominant descriptors, Write a review, Save beer, and Compare.
- Comparison: two human-readable beer selections, shareable slug state, same-beer prevention,
  technical and sensory tradeoffs, swap and clear actions, and comparison export.
- Discovery: country summaries and details, country beer rankings, style distribution, searchable
  style browsing, style details, related styles, and links back to individual beers.
- Product shell: global search, working settings, account navigation, Methodology, About, Contact,
  Data Counters, loading, empty, error, signed-out, and signed-in states.

When screenshot behavior is ambiguous, infer the most useful credible behavior, implement it fully,
and document the decision in `docs/SCREENSHOT-FUNCTIONALITY.md` or the owning architecture doc. Do
not preserve a control that does nothing.

## Stack and commands

- Next.js 16.2.10 App Router and React 19.2.4.
- TypeScript in strict mode. `@/*` maps to `src/*`.
- Supabase Postgres and Auth through `@supabase/supabase-js` and `@supabase/ssr`.
- Zod 4 for URL and mutation validation.
- Vitest 4 in the Node environment.
- Tailwind CSS 4 is loaded through PostCSS; the implemented interface is primarily in
  `src/app/globals.css`.
- Outfit is loaded with `next/font`. Lucide is the only interface icon library. GSAP is limited to
  the discovery experience.

Repository commands:

- `npm run dev` - start the Next.js development server.
- `npm run build` - create a production build.
- `npm run start` - serve the production build.
- `npm run lint` - run ESLint.
- `npm run typecheck` - run strict TypeScript without emitting files.
- `npm run test` or `npm test` - run Vitest once.
- `npm run test:watch` - run Vitest in watch mode.
- `npm run db:start` - start local Supabase when the Supabase CLI is available.
- `npm run db:reset` - recreate the local database, apply migrations, and load `supabase/seed.sql`.
- `npm run test:db` - run pgTAP database tests against a running local stack.
- `npm run db:types` - regenerate `src/lib/supabase/database.types.ts` from the local schema.

The Supabase CLI is not pinned in `package.json` at the time of this guide. The `db:*` scripts only
work when a compatible `supabase` command is on `PATH`. The README uses `npx supabase ...` for local
startup. Do not report a database command as passed unless it actually ran. If reproducible database
tooling is added, pin it and update `package.json`, `README.md`, and the engineering docs together.

## Agent workflow

1. Understand and reproduce the behavior before changing it.
2. Fix the owning layer, not the layer where a symptom happens to appear.
3. Prefer the smallest correct system-wide change, not the smallest diff.
4. Do not add fallbacks that hide internal contract failures. Fallbacks are for external
   uncertainty such as networks, user input, or third-party systems.
5. Match scope to the problem. Cross-layer failures require producer and consumer changes; local
   failures should stay local.
6. Inspect both sides of contracts involving data, auth, routing, caching, and asynchronous work.
7. Prefer clear local code over an abstraction that does not remove real complexity.
8. Act autonomously when a decision is safe, reversible, and within scope. Ask before destructive,
   product-defining, or ambiguous changes.
9. Stabilize severe user impact when needed, then document and complete the owning fix.
10. Make every change verifiable with tests or a clear reproducible check.

Before editing:

- Read the closest product, design, architecture, and Next.js documentation.
- Inspect `git status` and preserve unrelated user changes.
- Trace the current producer, transformation, and consumer instead of assuming the source of a bug.

Before finishing:

- Review `git diff` and `git status`.
- Run checks proportional to risk. Do not commit failing work.
- Update documentation for changes to architecture, schema, routes, auth, deployment, design tokens,
  commands, or workflow.
- **A commit is mandatory when the task changes repository-tracked files.** Stage and commit all and
  only the intended files before sending the final response, unless the user explicitly says not to
  commit. Do not leave completed work merely unstaged or uncommitted.
- Use a professional, emoji-prefixed commit subject, for example
  `📝 Add BeerBoard agent operating guide`.
- After committing, run `git status` again and report the commit hash and subject. If a commit cannot
  be created, state the reason clearly instead of presenting the work as complete.

## Project map and ownership

- `src/app/` owns App Router pages, metadata, route-level loading/error states, Server Actions, and
  CSV route handlers.
- Public routes are `/`, `/beers`, `/beers/[slug]`, `/compare`, `/countries`,
  `/countries/[slug]`, `/styles`, `/styles/[slug]`, `/search`, `/methodology`, `/about`, `/contact`,
  and `/data-counters`.
- Auth routes are `/login`, `/signup`, `/forgot-password`, `/reset-password`, `/auth/callback`, and
  `/auth/signout`. `/reviews`, `/saved`, and `/settings` are protected. `/ledger` redirects to `/saved`.
- `src/components/` owns presentation and small interaction islands. Current client islands are
  account state, mobile navigation, review and saved-beer actions, export feedback, theme control, and the GSAP
  discovery experience.
- `src/lib/ranking.ts` owns ranking URL parsing, validation, normalization, serialization, search
  sanitization, and the 10-row page size.
- `src/lib/comparison.ts` owns numeric comparison states and deterministic tradeoff notes.
- `src/lib/csv.ts` owns CSV cell escaping and missing-value text.
- `src/lib/catalog.ts` is the server-only public catalog repository. It owns Supabase queries,
  filtering, ordering, pagination, descriptor loading, and catalog aggregation used by routes.
- Country aggregation is mainly owned by the `country_summaries` SQL view. Country style
  distribution is currently derived from selected country beers in the country detail route.
- Style aggregation is owned by the `style_summaries` SQL view. Style search/grouping is currently
  performed by the style route after loading the summaries.
- `src/lib/supabase/public.ts` creates the browser-safe, unauthenticated server catalog client.
  `server.ts` creates cookie-backed server clients. `client.ts` creates the browser client.
  `proxy.ts` adapts auth-cookie refresh responses.
- `src/app/actions/reviews.ts`, `src/app/actions/saved.ts`, `src/app/actions/profile.ts`, and `src/app/auth/actions.ts` own
  validated mutations. `src/app/api/export/*` owns public CSV responses.
- `src/lib/supabase/database.types.ts` is generated from the applied schema. Handwritten helper
  aliases at its end narrow view fields guaranteed non-null by inner-join SQL contracts.
- `supabase/migrations/` owns schema history. `supabase/seed.sql` owns the repeatable fictional
  50-beer local dataset. `supabase/tests/rls.sql` owns pgTAP ownership checks.
- Unit tests are colocated as `src/lib/*.test.ts`.
- `DESIGN.md` and the root variables in `src/app/globals.css` own the design system. Global CSS is
  the current styling implementation; no CSS Modules exist yet.

Business behavior belongs in pure domain modules or PostgreSQL, whichever owns the invariant.
React components must not own ranking, scoring, filtering, aggregation, comparison, or CSV rules.
Server Components are the default. Add `"use client"` only when browser interaction requires it.
Validate external and mutation input with Zod. Fix contract failures in the owning layer.

## Data integrity and scoring

### Real schema

- `dataset_releases` owns the release title, audit time, demonstration flag, and disclosure.
- `countries` and `styles` own taxonomies and educational summaries.
- `breweries` belongs to a country.
- `beers` belongs to a release, brewery, country, and style. Technical measurements are nullable.
- `beer_assessments` is one-to-one with a beer. It owns score inputs, sensory dimensions, editorial
  verdict, methodology note, assessment time, and generated Index score.
- `descriptors` and `beer_descriptors` form a normalized many-to-many sensory vocabulary with
  intensity.
- `profiles.id` is the matching `auth.users.id` and owns display name and theme.
- `reviews` stores one public 1–5 rating and written review per user and beer.
- `ledger_entries` is the legacy internal table name for private saved beers. Its composite
  `(user_id, beer_id)` primary key prevents duplicates for one user.
- `beer_catalog`, `country_summaries`, and `style_summaries` are `security_invoker` views.

### Canonical score

`beer_assessments.index_score` is a stored generated column created by
`20260714203117_create_beerboard_catalog.sql`:

```text
quality × 35%
+ balance × 25%
+ distinctiveness × 20%
+ technical execution × 20%
```

This SQL expression is the only scoring implementation. TypeScript, CSV, and UI consumers read the
generated value. Any formula change requires one migration, migration verification, regenerated
types, methodology documentation, and score/ranking tests.

### Data rules

- Stable database IDs are identity. Slugs are human-readable URL labels, not identity.
- Derive rank from score. The `beer_catalog` view currently orders by `index_score DESC, name ASC`
  for stable tie-breaking. Do not seed or hardcode ranks.
- Derive counts, means, brewery coverage, leading beers, style distributions, and freshness from
  database data. Do not hardcode values that can be calculated.
- Do not fabricate beer claims, freshness dates, brewery counts, editorial evidence, or methodology
  results.
- Distinguish measured technical values, editorial assessments, and derived values in schema, code,
  and copy.
- Missing measurements stay `null`. Render the exact text **Not reported**, never `0`, a fabricated
  estimate, or an unexplained dash.
- Comparison summaries derive only from the two selected beers. Describe tradeoffs and never declare
  an overall winner when the evidence does not support one.
- Keep demonstration disclosure attached to the current release and visible in the relevant UI.
- Keep generated types synchronized with the schema actually deployed to BeerBoard.
- Known implementation gaps are recorded in `docs/ARCHITECTURE.md`. Do not normalize those gaps into
  a new contract; fix them in the owning data or presentation layer.

## Supabase security contract

- Enable RLS on every table in an exposed schema.
- Public catalog tables may have explicit `SELECT` policies for `anon` and `authenticated`. They must
  not be writable by normal browser users.
- Owner-specific policies must compare ownership with `auth.uid()`. `TO authenticated` alone is not
  authorization.
- Update policies need both `USING` and `WITH CHECK`. Insert policies need ownership in
  `WITH CHECK`; delete/select policies need ownership in `USING`.
- Never authorize from `user_metadata`. It is user-editable and may be used only for harmless display
  or onboarding defaults. Authorization roles and permissions must come from server-controlled data
  such as `app_metadata`, signed claims, or owner-scoped rows.
- Server authorization uses `auth.getUser()`, not `getSession()` or a client-provided user ID.
- Browser Supabase code uses only the publishable or compatible anonymous key plus RLS.
- Never expose the service-role key or any secret key to browser code. Never put secrets in
  `NEXT_PUBLIC_*` variables.
- Data API views must preserve underlying RLS with `security_invoker = true` where appropriate.
- Avoid `SECURITY DEFINER`. If it is genuinely required, place it in a non-exposed schema where
  possible, set a safe search path, revoke default execution, perform explicit authorization, and
  document the reason.
- `private.handle_new_user()` creates a profile from a harmless display-name default.
- `private.set_review_author()` copies the server-owned profile display name onto reviews so clients
  cannot choose another author name. Both security-definer functions have safe search paths and
  execution revoked from public browser roles.
- Privileged writes must use validated server-only routes, actions, or narrowly scoped functions.
- Every owner-specific feature needs a positive owner test and a negative cross-user test.
  `supabase/tests/rls.sql` covers profiles, reviews, and saved-beer rows.
- Authentication redirects must work on localhost, Vercel previews, and production.
- Verify the Supabase project URL/reference before every remote migration. Never apply BeerBoard
  migrations to the EasyPrompt project. Do not infer a target from a nearby checkout.

## Database workflow

For every schema, policy, view, trigger, index, or database-function change:

1. Inspect and explicitly identify the target BeerBoard Supabase project.
2. Check which local and remote migrations are already applied.
3. Generate a timestamped migration through the established Supabase CLI workflow.
4. Review the SQL for destructive statements, lock risk, unsafe grants, missing RLS, and unsafe
   function search paths. Remember that `supabase/seed.sql` intentionally truncates local data.
5. Apply the migration to the confirmed target before matching application code is complete.
6. Run post-migration queries that verify columns, constraints, views, grants, policies, and expected
   row behavior.
7. Test RLS-positive owner cases and RLS-negative anonymous/cross-user cases.
8. Run available Supabase database security and performance advisors and resolve relevant findings.
9. Regenerate `src/lib/supabase/database.types.ts` from the applied schema and review its diff.
10. Update `docs/ARCHITECTURE.md` and operational documentation.

Adding a migration file without applying and verifying it is incomplete work. Never hide a failed
migration behind application fallbacks. Inspect both SQL producers and TypeScript/UI consumers when
a view or column contract changes.

## Authentication architecture

- Supabase Auth uses secure cookie-backed SSR through `@supabase/ssr`.
- `src/lib/supabase/server.ts` owns cookie-aware server clients. Protected pages and actions validate
  the user with `auth.getUser()`.
- `src/proxy.ts` refreshes claims only for reviews, saved beers, settings, auth, login, and signup routes. Keep the
  matcher focused.
- Public catalog reads use `src/lib/supabase/public.ts`, which has no persisted session, refresh, URL
  session detection, or cookie access.
- Do not read user state in the root layout. `AccountMenu` is the focused client boundary that loads
  account display state after mount.
- Keep public routes cacheable where possible. Current public catalog routes use one-hour
  revalidation; reviews, saved beers, settings, and auth-purpose pages are forced dynamic.
- Review management, saved beers, and settings must enforce authentication on the server. Client gates are only UX and never
  authorization.
- Signed-out users may see the real Write a review and Save beer actions. After intent, send them to authentication
  with a safe return path. Preserve the intended beer through login or signup.
- Validate return paths as local absolute paths. Never allow protocol-relative or external redirects.
- Reading `user_metadata.display_name` for display is not authorization. Keep profile ownership and
  permissions in server-controlled rows and RLS.

## Vercel deployment contract

Vercel is the required deployment target. This checkout is locally linked to the Vercel project
`beerboard`; `.vercel/` is local and gitignored. There is no `vercel.json`, so the application uses
the standard Vercel Next.js build behavior. There is no repository deployment script.

Required application variables for Preview and Production:

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`
- `NEXT_PUBLIC_SITE_URL`

`NEXT_PUBLIC_VERCEL_URL` may be supplied automatically by Vercel for previews. Public variables are
inlined at build time, so they must exist in the build environment. Keep `.env.example` synchronized
with required application configuration and never add values or secrets. Keep service keys, tokens,
and any future privileged credentials server-only; the current application does not require a
service-role key.

Deployment order:

1. Confirm the BeerBoard Supabase project and apply verified migrations before dependent code.
2. Regenerate database types and run migration, RLS, advisor, lint, type-check, unit-test, and build
   checks.
3. Verify Vercel environment values for the target environment, including the preview or production
   site URL used to construct auth callbacks.
4. Use Git-connected deployments when available. Create a preview before production.
5. Inspect build logs and final deployment status. Test the deployed public catalog and critical
   owner flow.
6. Verify Supabase Auth allows localhost, the Vercel preview pattern, the production origin, and the
   `/auth/callback` return flow.
7. Return the live deployment URL and status. Do not call a deployment successful without both.

## UI and design rules

- BeerBoard must feel credible, legible, curious, welcoming, and task-focused. Lead with beer, maker,
  place, style, and taste. Keep technical evidence available without making a dashboard.
- Screenshots are functional references, not visual targets. Do not copy weak typography,
  spreadsheet-first layouts, or desktop-only composition.
- Use Outfit for display, body, controls, and measurements. Use tabular numerals for scores, ranks,
  dates, and measurements. Display typography does not belong in controls or data labels.
- Use the implemented neutral background, amber discovery/action, and bottle-green trust palette.
  Reuse root color, radius, shadow, content-width, and easing variables from `globals.css` and the
  values documented in `DESIGN.md`.
- The current CSS does not define a complete shared spacing or z-index token scale. Do not invent
  component-local systems. Add reusable tokens centrally before repeating new spacing or elevation
  values, and document a design-system change.
- Do not introduce raw component-local colors when an existing semantic token fits.
- Use Lucide only. Do not mix icon systems, use emoji as interface icons, or add hand-drawn SVG
  controls.
- Do not add fake settings, account, filter, search, export, comparison, review, or saved-beer controls.
- Prefer open editorial sections, ranked lists, and purposeful imagery. Avoid nested cards and
  repetitive card grids.
- Avoid gradient text, decorative glassmorphism, decorative badges, excessive rounding, oversized
  rounded containers, colored side stripes, and arbitrary shadows.
- Do not put tiny uppercase eyebrow text above every section.
- Body text must meet WCAG 2.2 AA contrast. Focus states must remain visible and state must not rely
  on color alone.
- Touch targets must be at least 44px on coarse pointers. Use semantic controls and clear labels.
- Support 320px and wider without page-level horizontal overflow. Mobile layouts must recompose,
  not shrink desktop layouts.
- Motion must explain hierarchy or state. Keep comparison, auth, settings, reviews, and saved beers calm. Respect
  `prefers-reduced-motion`.
- Data visualizations require exact accessible text or table/list alternatives.
- Test long beer, brewery, style, and country names, null values, and empty results.

App Router CSS is global even when imported from a route. `src/app/globals.css` currently contains
the shared global implementation. Prefer CSS Modules for isolated new work. If route-global CSS is
unavoidable, prefix selectors beneath a unique page-root class. Never introduce generic route
selectors such as `.card` or `.layout` that can collide elsewhere.

## Testing and verification

For broad or release-bound work, the quality gate is:

```powershell
npm run lint
npm run typecheck
npm run test
npm run test:db
npm run build
```

`npm run test:db` requires a running local Supabase stack and an available Supabase CLI. If that
prerequisite is missing, report the database check as not run; do not call it passed. Database and
deployment checks are mandatory when their owning layers change.

Current automated coverage:

- `src/lib/ranking.test.ts` covers filter parsing, defaults, URL serialization, and search
  sanitization.
- `src/lib/comparison.test.ts` covers comparison states and deterministic no-winner notes.
- `src/lib/csv.test.ts` covers quoting, quote escaping, null text, and the UTF-8 BOM.
- `supabase/tests/rls.sql` covers public review reads, review ownership, private saved beers, and profiles.

Critical domain coverage to preserve or add when affected:

- Catalog schema invariants and unique stable IDs/slugs.
- Canonical score calculation and generated-column boundaries.
- Stable ranking and score-tie ordering.
- Search across beer, brewery, country, and style.
- Combined country, style, minimum-score, search, sort, and pagination filters.
- Ranking URL-state parse/serialize round trips.
- Country and style aggregates, including empty datasets.
- Missing measurements remaining null and rendering as **Not reported**.
- Comparison generation, unavailable metrics, and same-beer prevention.
- CSV delimiters, quotes, newlines, Unicode, nulls, and export filter parity.
- Review public reads, author ownership, owner mutations, and cross-user denial.
- Saved-beer ownership, duplicate prevention, owner mutations, and cross-user denial.
- Profile ownership and cross-user denial.

Do not launch a dev server or browser for every UI change. Use code inspection and targeted checks for
small or medium UI work. Use browser verification for broad, high-risk, interaction-heavy changes
where rendered behavior cannot be reasoned about safely. For broad UI work, check desktop, tablet,
and mobile widths, keyboard flow, reduced motion, signed-in/out states, and horizontal overflow.
